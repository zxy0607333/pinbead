## 当前结论

Pinbead 第一阶段应优先做英文工具站和原创图纸库，不急着做完整后端、用户系统和社区。图片转换优先在浏览器本地完成；公开图纸库先使用站点自有原创或授权安全内容；AdSense 放到内容页和教程页为主，避免干扰上传、生成、下载这些核心操作。

## 1. V0.1 快速上线版

目标：快速上线一个可用、可收录、可继续扩展的英文站。这个版本不做完整后端，不做用户登录，不做社区投稿。

### [~] 1.1 冻结 V0.1 功能范围

需求：

- 明确第一版只做英文站点、图片转拼豆图纸工具、基础原创图纸库、基础 SEO 页面和合规页面。
- 明确第一版不做登录、用户保存、公开投稿、点赞、评论、收藏、付费系统和复杂后台。
- 明确图片处理默认在浏览器本地完成，不把用户原图上传到服务器。
- 明确第一版图纸数据可以先用代码仓库里的静态数据维护，例如 TypeScript、JSON 或 MDX。
- 明确第一版部署到自有服务器，可使用 Next.js standalone 或静态导出，具体方式等代码启动后再定。

验收标准：

- V0.1 范围在文档中清晰可见。
- 后续开发任务不会把社区、登录、数据库误认为首发必需项。

### [x] 1.2 初始化前端项目

需求：

- 使用 Next.js App Router 初始化项目。
- 使用 TypeScript。
- 使用 Tailwind CSS。
- 建立基础目录结构，例如 `app`、`components`、`lib`、`data`、`public`。
- 配置基础代码规范和构建脚本。
- 确保项目能在本地通过 `npm run dev` 或等效命令启动。
- 确保项目能通过生产构建命令生成可部署产物。

验收标准：

- 本地开发服务能正常启动。
- 首页能访问。
- 生产构建能通过。
- 仓库中不提交构建产物和依赖目录。

### [ ] 1.3 建立英文首页

需求：

- 首页首屏直接展示图片转拼豆图纸工具入口，不做空泛营销页。
- 页面品牌显示为 `Pinbead`。
- H1 建议围绕 `Pinbead Pattern Maker` 或 `Image to Bead Pattern Maker`。
- 副标题说明用户可以把图片转换成 printable pin bead pattern。
- 首屏要有上传入口、简短隐私提示和主要设置入口。
- 首页下方展示精选原创图纸、热门分类和入门教程链接。
- 移动端首屏不能被过多文字或广告挤压。

验收标准：

- 用户进入首页后能立即理解站点用途。
- 首页可以引导用户上传图片或浏览图纸库。
- 页面标题、描述和可见文案适合英文 SEO。

### [ ] 1.4 实现浏览器端图片上传与预览

需求：

- 支持用户从本地选择图片文件。
- 支持常见格式，例如 PNG、JPEG、WebP。
- 上传后在浏览器本地生成预览。
- 对过大的图片进行前端尺寸限制或压缩处理，避免浏览器卡死。
- 需要显示明确提示：图片默认只在浏览器本地处理，不会自动上传到服务器。
- 不在第一版保存用户原图。

验收标准：

- 用户能选择图片并看到预览。
- 大图不会导致页面明显崩溃。
- 刷新页面后用户原图不会被服务器保存。

### [ ] 1.5 实现裁剪和尺寸设置

需求：

- 支持基础裁剪，至少提供正方形裁剪和保持原比例两种方式。
- 支持设置输出格子尺寸，例如 16x16、24x24、32x32、48x48、64x64。
- 尺寸设置要说明越大越精细，但制作难度和用豆数量也越高。
- 移动端操作要可用，控件不能过小。
- 第一版可以不做复杂自由变形和多区域裁剪。

验收标准：

- 用户可以控制图纸宽高或选择预设尺寸。
- 生成结果与用户选择的尺寸一致。
- 裁剪区域在桌面和移动端都能正常操作。

### [ ] 1.6 实现像素化转换

需求：

- 使用 Canvas 或 OffscreenCanvas 把图片缩放到目标格子尺寸。
- 每个输出格子对应一个像素色块。
- 生成过程要尽量流畅，较大的转换任务应放到 Web Worker。
- 第一版至少要支持基础像素化，不必一开始实现所有高级算法。
- 对透明背景需要有策略，例如转换为白色、保留透明或让用户选择背景色。

验收标准：

- 用户能得到清晰的像素图预览。
- 目标尺寸越小，像素化效果越明显。
- 转换过程不会长时间阻塞 UI。

### [ ] 1.7 实现拼豆色卡匹配

需求：

- 建立至少一个基础品牌色卡，建议先从 Perler 或通用色卡开始。
- 每个颜色需要包含名称、HEX、可选品牌色号。
- 将像素颜色映射到最接近的拼豆颜色。
- 色彩匹配建议使用 LAB 色彩空间或预留 LAB 计算能力，不建议长期只依赖 RGB 距离。
- UI 上允许用户选择色卡品牌，但第一版可以只开放一个默认色卡。

验收标准：

- 生成结果使用有限的拼豆颜色，而不是任意 RGB 颜色。
- 页面能展示颜色名称和 HEX。
- 后续可以扩展 Hama、Artkal、MARD 等色卡。

### [ ] 1.8 实现颜色数量限制

需求：

- 支持用户选择最大颜色数，例如 8、16、24、32。
- 颜色数量越少，图纸越适合初学者；颜色数量越多，图像还原越细腻。
- 第一版可以采用简单量化算法，后续再优化为更高质量的 k-means 或 median cut。
- 生成结果中使用的颜色数量不能超过用户选择上限。

验收标准：

- 用户调整颜色数量后，预览和颜色清单同步变化。
- 颜色数量上限生效。
- 低颜色模式下图纸仍然可识别。

### [ ] 1.9 实现图纸预览模式

需求：

- 至少支持普通像素格预览。
- 支持圆珠预览，用圆形 beads 视觉模拟实际拼豆效果。
- 支持网格线开关，方便用户对照制作。
- 预览区域需要响应式，桌面端清晰，移动端不溢出。
- 不要让预览区因为工具栏变化而频繁跳动。

验收标准：

- 用户可以在不同预览模式之间切换。
- 预览在移动端和桌面端都可读。
- 圆珠预览能直观表达拼豆成品效果。

### [ ] 1.10 实现颜色清单和用豆数量

需求：

- 根据生成图纸统计每种颜色使用的 bead 数量。
- 清单中显示颜色块、颜色名称、色号、HEX 和数量。
- 显示总 bead 数。
- 清单需要和当前预览结果实时对应。
- 后续要能用于 PDF 导出和详情页展示。

验收标准：

- 颜色清单数量总和等于图纸格子数量，透明格除外。
- 用户能根据清单准备材料。
- 调整尺寸或颜色数后清单自动更新。

### [ ] 1.11 实现 PNG 导出

需求：

- 支持导出当前图纸预览为 PNG。
- 导出图应包含清晰的图纸主体。
- 可以提供带网格和不带网格两种导出模式，第一版至少实现一种。
- 文件名应包含 `pinbead`、尺寸和时间或标题，方便用户识别。
- 导出过程在浏览器端完成。

验收标准：

- 用户点击下载后得到可打开的 PNG 文件。
- PNG 内容与页面预览一致。
- 下载功能在主流桌面浏览器可用。

### [ ] 1.12 建立基础原创图纸库

需求：

## 2. V0.2 内容与导出版

目标：增加内容厚度和导出能力，为 AdSense 审核和自然搜索增长做准备。

### [ ] 2.1 扩展原创图纸库到 50-80 个

需求：

- 优先制作 beginner-friendly 图纸。
- 分类覆盖 Animals、Food、Holidays、Cute Icons、Nature、Letters & Numbers、Keychain Patterns。
- 每个图纸必须原创或授权安全。
- 每个图纸需要保留源数据，便于后续重新生成预览、PDF 和色卡版本。
- 避免批量生成高度重复的薄内容页面。

验收标准：

- 公开图纸数量达到 50 个以上。
- 每个主要分类都有足够内容。
- 图纸页面质量足以支撑 AdSense 审核。

### [ ] 2.2 实现 PDF 导出

需求：

- 工具页支持把当前生成图纸导出为 PDF。
- PDF 至少包含图纸预览、尺寸、颜色清单和总 bead 数。
- 图纸详情页也要提供对应 PDF 下载。
- PDF 要适合 A4 或 Letter 打印。
- 大尺寸图纸需要考虑分页或缩放。

验收标准：

- 用户可以打印 PDF 并按图制作。
- PDF 中颜色清单清晰可读。
- PDF 下载文件名规范。

### [ ] 2.3 实现更完整的图纸详情页内容

需求：

- 每个图纸详情页增加 2-4 段英文制作说明。
- 说明内容要具体，例如适合新手、建议底板尺寸、适合做钥匙扣或装饰。
- 增加 related patterns。
- 增加分类、标签和难度入口。
- 避免所有图纸使用同一段模板化描述。

验收标准：

- 每个详情页有独特正文。
- 用户能从一个图纸继续浏览相关图纸。
- 页面不显得像自动生成的空壳 SEO 页面。

### [ ] 2.4 发布第一批教程文章

需求：

- 至少发布 3-5 篇英文教程。
- 优先主题包括：
  - How to Turn a Photo into a Bead Pattern
  - Beginner Pin Bead Tips
  - How to Choose the Right Bead Pattern Size
  - Bead Color Chart
  - How to Print and Follow a Bead Pattern
- 教程文章要自然链接到工具页和图纸库。
- 文章要使用真实截图、图纸示例或步骤图，避免纯文字堆砌。

验收标准：

- 教程页可以作为 AdSense 和 SEO 的主要内容承载。
- 文章能回答用户真实问题。
- 每篇文章都有清晰标题、描述和内链。

### [ ] 2.5 优化转换质量

需求：

- 改进颜色匹配，优先使用 LAB 色彩空间。
- 增加可选抖动开关。
- 增加轮廓增强或对比度调整选项。
- 增加背景色或透明背景处理选项。
- 需要保持默认设置简单，不让新手被高级选项吓到。

验收标准：

- 同一张图片生成效果比 V0.1 更接近可制作图纸。
- 高级选项不会破坏基础流程。
- 移动端仍然易用。

### [ ] 2.6 建立轻量内容录入方式

需求：

- 如果图纸仍使用静态数据，建立统一 schema。
- 提供校验脚本或构建时校验，避免缺少 slug、标题、分类、预览图等字段。
- 图纸源数据、预览图、下载文件的路径规则要统一。
- 为后续迁移到数据库预留字段，例如 status、sourceType、author。

验收标准：

- 新增图纸不需要改多个无关文件。
- 缺字段时构建或脚本能提醒。
- 后续迁移数据库时数据结构不会完全推倒重来。

## 3. V0.3 AdSense 准备与接入

### [ ] 3.1 申请前站点体检

需求：

- 确认站点没有空白页、测试页、明显重复页和未完成页面。
- 确认首页、工具页、图纸库、教程页、隐私政策、联系页面都能正常访问。
- 确认内容是原创或授权安全。
- 确认移动端体验良好。
- 确认页面加载速度和 Core Web Vitals 没有明显问题。

验收标准：

- 站点达到可提交 AdSense 审核状态。
- 不存在明显低质量或违规内容。

### [ ] 3.2 配置 ads.txt

需求：

- 根据 AdSense 后台提供的 publisher id 生成 ads.txt。
- ads.txt 必须部署在站点根路径。
- 部署后通过浏览器访问确认内容正确。

验收标准：

- `https://domain.com/ads.txt` 可访问。
- AdSense 后台不再提示 ads.txt 缺失或错误。

### [ ] 3.3 创建广告组件

需求：

- 封装统一 AdSense 组件。
- 支持按页面位置配置广告单元。
- 广告组件需要避免造成严重 CLS。
- 广告容器要有稳定尺寸或合理占位。
- 开发环境可以不加载真实广告。

验收标准：

- 广告代码集中管理。
- 页面不会因为广告加载明显跳动。
- 后续可以方便调整广告位置。

### [ ] 3.4 设置安全广告位

需求：

- 优先在图纸详情页、分类页和教程页放广告。
- 工具页可以少量测试，但不能放在上传、生成、下载按钮附近。
- 移动端首屏不要用广告挤压主功能。
- 未审核 UGC 页面不展示广告。
- 广告与内容要有清晰区分，不能诱导误点。

验收标准：

- 广告位符合 AdSense 体验和政策要求。
- 工具核心流程不被广告打断。
- 页面布局在桌面和移动端都稳定。

### [ ] 3.5 完善隐私和同意管理预留

需求：

- 隐私政策说明 cookies、广告标识符、Google Analytics、AdSense 和第三方服务。
- 面向 EEA、英国、瑞士等地区用户时，预留 CMP 或 cookie consent 接入能力。
- 说明用户上传图片默认本地处理。
- 不把用户生成内容用于未声明用途。

验收标准：

- 隐私政策覆盖广告和统计场景。
- 后续接入正式 CMP 不需要重构全站。

## 4. V0.4 用户保存与分享

目标：当自然流量和用户需求出现后，再开放用户系统和分享能力。

### [ ] 4.1 判断是否启动用户系统

需求：

- 不以主观想象启动社区功能，要看实际指标。
- 参考触发条件：
  - 月自然搜索访问超过 1,000。
  - 工具页持续有生成和下载事件。
  - 用户通过邮件、社媒或反馈表达保存/分享需求。
  - 图纸库页面有稳定浏览深度。
- 如果指标不足，继续优先做内容和工具体验。

验收标准：

- 有数据支撑是否进入 V0.4。
- 不在流量不足时过早引入审核、账号和社区成本。

### [ ] 4.2 接入用户登录

需求：

- 使用 Auth.js 或同类自托管友好方案。
- 首期支持 Google OAuth 和邮箱登录。
- 用户登录后可以保存自己的图纸。
- 登录不是使用工具的前置条件。
- 登录流程要简洁，不能阻碍图片生成和下载。

验收标准：

- 用户可以登录和退出。
- 匿名用户仍能使用核心工具。
- 用户身份可以关联保存的图纸。

### [ ] 4.3 增加数据库

需求：

- 使用 PostgreSQL。
- 使用 Prisma 管理 schema 和迁移。
- 存储用户、图纸元数据、分类、标签、收藏、审核状态等业务数据。
- 不把每个像素格作为数据库行存储。
- 图纸矩阵建议用压缩 JSON 存对象存储或文本字段。

验收标准：

- 数据库迁移可重复执行。
- 基础数据模型支持用户保存和公开图纸。
- 不引入明显低效的数据结构。

### [ ] 4.4 接入对象存储

需求：

- 使用 Cloudflare R2 或兼容 S3 的对象存储。
- 存储公开预览图、缩略图、PDF 和 pattern JSON。
- 不长期保存用户未公开的原图，除非后续有明确需求和隐私说明。
- 为临时文件设置生命周期清理策略。

验收标准：

- 公开图纸资产可以稳定访问。
- 存储路径规则清晰。
- 未发布或被拒绝内容可以清理。

### [ ] 4.5 实现私有保存

需求：

- 登录用户可以把生成结果保存到个人空间。
- 保存内容包括图纸元数据、颜色清单、预览图和 pattern JSON。
- 私有图纸默认不公开、不进 sitemap、不展示广告。
- 用户可以删除自己的私有图纸。

验收标准：

- 用户能保存和再次打开图纸。
- 私有图纸无法被未授权用户访问。
- 删除操作能清理关联资产或标记为删除。

### [ ] 4.6 实现公开分享申请

需求：

- 用户可以从私有图纸点击 `Share to gallery`。
- 分享前必须确认拥有图片或图纸权利，并授权 Pinbead 公开展示。
- 提交后进入审核队列。
- 审核前页面不公开、不索引、不展示广告。
- 用户可以看到提交状态。

验收标准：

- 分享行为有明确授权确认。
- 未审核内容不会进入公开图库。
- 管理员能看到待审核内容。

### [ ] 4.7 建立审核后台

需求：

- 管理员可以查看待审核图纸。
- 管理员可以批准、拒绝、下架图纸。
- 审核页面要显示预览图、标题、描述、来源、用户和提交时间。
- 拒绝时可以记录原因。
- 审核通过后生成公开页面并允许收录。

验收标准：

- 审核流程可闭环。
- 违规内容可以快速下架。
- 审核状态清晰记录。

### [ ] 4.8 增加举报机制

需求：

- 公开图纸页提供 Report 入口。
- 举报原因包括版权、成人内容、暴力仇恨、垃圾内容、其他。
- 举报进入后台处理。
- 被多次举报的内容可以临时下架或进入复审。

验收标准：

- 用户能举报公开内容。
- 管理员能处理举报。
- 高风险内容有快速止损能力。

## 5. V1.0 社区与规模化

目标：当站点已有稳定自然流量和投稿需求后，增强社区和商业化能力。

### [ ] 5.1 建立公开 Gallery

需求：

- 展示审核通过的用户分享图纸。
- 支持分类、标签、难度、尺寸筛选。
- 支持按最新、热门、下载量排序。
- Gallery 页面要避免低质量重复内容占满首页。
- 需要继续保持版权和安全审核。

验收标准：

- 用户可以浏览社区作品。
- Gallery 不影响原创图纸库质量。
- 搜索引擎只收录审核通过的内容。

### [ ] 5.2 实现收藏和点赞

需求：

- 登录用户可以收藏图纸。
- 登录用户可以点赞公开图纸。
- 点赞和收藏不要成为首屏核心功能，避免干扰下载和制作。
- 需要防止重复点赞和刷量。

验收标准：

- 用户可以在个人页找到收藏内容。
- 图纸详情页能显示基础热度。
- 数据不会被匿名刷量轻易污染。

### [ ] 5.3 建立用户个人页

需求：

- 用户可以展示审核通过的公开图纸。
- 用户可以管理自己的私有图纸和公开投稿。
- 个人页不展示未审核内容给其他用户。
- 用户资料需要基本反滥用规则，例如昵称长度、头像安全、简介链接限制。

验收标准：

- 作者页可以提升社区感。
- 个人页不会成为垃圾链接入口。
- 用户能管理自己的内容。

### [ ] 5.4 增强搜索

需求：

- 初期可以使用 PostgreSQL 全文搜索。
- 内容量增大后再考虑 Meilisearch、Typesense 或 Algolia。
- 搜索字段包括标题、描述、分类、标签、尺寸、难度。
- 搜索结果要优先展示高质量和审核通过内容。

验收标准：

- 用户能快速找到图纸。
- 搜索不会返回未审核或私有内容。
- 搜索性能满足当前内容规模。

### [ ] 5.5 商业化扩展

需求：

- 在 AdSense 之外探索耗材 affiliate。
- 可测试高级 PDF 模板、无水印导出、批量转换和教师活动包。
- 付费功能不能破坏免费工具的基础价值。
- 商业化页面要符合广告、支付和消费者保护要求。

验收标准：

- 至少有一个 AdSense 之外的收益实验。
- 免费用户仍能完成核心转换和下载。
- 付费功能有清晰价值，而不是人为制造障碍。

## 6. 运维、安全与质量

### [ ] 6.1 建立生产部署流程

需求：

- 明确自有服务器部署命令和目录。
- 使用 Docker Compose 或 systemd 管理服务。
- 配置 Nginx 或 Caddy 反向代理。
- 配置 HTTPS 证书自动续期。
- 建立回滚方式。

验收标准：

- 新版本可以稳定部署。
- 部署失败时可以回滚。
- 服务重启后能自动恢复。

### [ ] 6.2 建立备份策略

需求：

- 如果使用 PostgreSQL，至少每日备份。
- 备份保留 7-14 天。
- 重要对象存储资产要有恢复策略。
- 定期测试备份可用性。

验收标准：

- 数据库损坏时可以恢复。
- 备份文件不和生产密钥混放。
- 有明确恢复步骤。

### [ ] 6.3 建立错误监控

需求：

- 接入 Sentry 或同类工具。
- 前端错误和服务端错误都要记录。
- 图片转换失败、导出失败、页面渲染失败需要可追踪。
- 不上传用户原图作为错误上下文。

验收标准：

- 线上异常能被发现。
- 错误信息不会泄露用户隐私。
- 关键路径异常有足够上下文排查。

### [ ] 6.4 建立性能检查

需求：

- 上线前检查 Lighthouse。
- 重点关注移动端性能、CLS、LCP、图片尺寸和广告位布局。
- 图纸预览和转换过程不能明显卡顿。
- 静态资源要正确缓存。

验收标准：

- 移动端核心页面体验可接受。
- 广告和图片不会导致严重布局偏移。
- 用户可以顺畅完成上传、生成和下载。

### [ ] 6.5 建立反滥用策略

需求：

- 在用户投稿、登录、举报等功能上线后加入限流。
- 使用 Cloudflare Turnstile 或同类方案保护高风险表单。
- 限制垃圾链接、重复提交、批量注册。
- 对公开 UGC 保留快速下架能力。

验收标准：

- 表单不容易被批量滥用。
- 垃圾内容不会直接进入公开页面。
- 管理员能快速处理异常内容。

## 7. 暂缓功能

这些功能不是不要做，而是流量没起来前不建议优先做。

### [ ] 7.1 暂缓评论系统

需求：

- 评论会带来审核、垃圾内容和政策风险。
- 在没有稳定访问和社区需求前，不实现评论。
- 后续如果做评论，必须有举报、审核、屏蔽和反垃圾机制。

验收标准：

- V0.1 到 V0.4 不把评论作为必需功能。

### [ ] 7.2 暂缓站内私信和关注流

需求：

- 私信和关注流会显著增加社区治理复杂度。
- 只有在有大量创作者和重复访问用户后再考虑。
- 前期通过收藏、作者页和公开 Gallery 已经足够。

验收标准：

- 不在早期版本引入私信、关注动态和通知中心。

### [ ] 7.3 暂缓付费会员

需求：

- 早期优先验证工具需求、SEO 流量和 AdSense 收益。
- 付费功能应建立在明确用户痛点上，例如高级 PDF、批量转换、教师包。
- 不用会员墙阻碍基础转换功能。

验收标准：

- V0.1 到 V0.3 不做会员。
- 后续付费实验有明确用户价值。

### [ ] 7.4 暂缓移动 App

需求：

- 当前优先做移动网页体验。
- App 会增加开发、审核和维护成本。
- 如果未来有高频制作、离线使用或进度追踪需求，再评估 PWA 或原生 App。

验收标准：

- 早期只保证响应式网页体验。

### [ ] 7.5 暂缓全自动 AI 审核

需求：

- 早期用户投稿量不足时，人工审核更可控。
- AI 审核可以后续作为辅助，不应替代最终人工判断。
- 高风险版权内容仍需要人工判断。

验收标准：

- V0.4 首版分享功能使用人工审核为主。
